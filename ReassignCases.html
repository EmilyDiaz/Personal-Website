Overview

Problem

Without proper guardrails in place, any user could change issues and specific details of an issue, which could lead to a case being misrouted or mismanaged as well as incorrect reporting.

Challenges faced

  • Last minute changes from stakeholders, albeit minor, kept the solutioning team pivoting and on our toes.
  • Working with multiple designers at a time in the same file, while managing communication issues during PTO and other events for all designers.

Timeline

April 2023-July 2024

Role

UX Designer

Solution breakdown

Expand on existing functionality by adding new request and review pathways for the addition, removal, and modification of issues on cases.

Process

1. Mapping out the user flow

We approached the requirements gathering phase by meeting up with the stakeholders, getting a better understanding of the ask, and then exploring the existing functionality internally amongst ourselves. From there, we continued with our weekly cadence of stakeholder meetings to continue asking more questions and presenting initial wireframes until we were able to flesh both requirements and wireframes out enough. 2. Admin Flow

The admin users identified during stakeholder meetings would be the users that review an issue addition, modification, and removal request for a specific case. These users would go into the typical team queue, which would have a new column that holds these requests.

From there, the admins can click on a specific request and review the proposed changes made by a regular user. A case with more than one request could not be moved into the completed stage of the case lifecycle until all requests were reviewed (either approved or rejected) by an admin user. If a claim had several requests on it, each one could potentially be reviewed by different admin users.

The non-admin users identified during stakeholder meetings would be the users that are reviewing the claims on a daily basis. If deemed necessary, these users would be the ones to submit an issue addition, modiication, or removal request on a specific case that was in a completed status.

Cases that were in an "in progress" state could not have requests added to them by non-admin users, though admin users could still come in and add, remove, or modify issues in the regular manner (no request system). An additional use case did exist for non-admin users, where they could go into a claim with pending requests and add additional requests. A read-only state had to be established for prior requests that were submitted by someone other than the visiting user, while still allowing that visting user to:

  • Add new requests onto the claim
  • Edit their previous requests on the claim, if any
  • Cancel their previous request on the claim, if any

Reflections

I was the main point of contact for the UX design team for the majority of this project's lifecycle. However, this project did shift hands quite a bit within the UX team, and for a good portion of it I worked alongside three other designers. Having to work with multiple designers was not entirely a new experience for me, but it was still a learning experience with pros and cons. I learned a bit more about my own process watching other designers work, and what things I could do to not only my own design methods and efficient but also how to better communicate with developers during handoff.

One of the things that was introduced to me in this effort was the idea of annotated designs, where as UX designers we could mark and lay down notes on design specs and functionality that we wanted to highlight for the developers. Ever since then, I have been using this method of communication with developers and have expanded upon it by making it more of a dev guide, providing content outlines that are interactable for easy navigation.

However, too many moving hands can also cause issues, such as miscommunication in the design process and preferences being put above others. I learned to express what I was and was not comfortable with in terms of work methods, and establish boundaries with my teammates, such as not wanting to push last minute changes before I was about to present.
